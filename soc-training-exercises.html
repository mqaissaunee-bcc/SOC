1 Alert Severity Assessment & Prioritization Beginner ⏱ 20 min ▼

Learning Objectives

  • Understand alert severity levels and their meanings
  • Practice prioritizing alerts based on severity and context
  • Learn to filter alerts effectively in a SIEM dashboard

Screens Used

Screen 1: SIEM Dashboard

Scenario

You have just started your shift as a Tier 1 SOC analyst. The SIEM dashboard shows 47 unacknowledged alerts that accumulated overnight. You need to triage these alerts and determine the order in which they should be investigated.

Tasks

  • Open Screen 1 (SIEM Dashboard) and observe the alert queue for 2 minutes
  • Use the severity filters to view only Critical alerts. Document how many critical alerts exist.
  • For the following alert types, identify the appropriate severity level and explain why:

    Alert A: Failed SSH login from internal IP 10.0.1.45

    Alert B: Ransomware signature detected on endpoint WKS-042

    Alert C: Unusual DNS query to newly registered domain

    Alert D: Data exfiltration detected - 500MB upload to external IP

  • Create a prioritized list ranking these 4 alerts from highest to lowest priority
  • Explain the factors that influenced your prioritization decisions
🔒 View Answer Key ▼
Severity Classifications:
  • Alert A (Low/Medium): Failed SSH login from internal IP - could be typo, but monitor for pattern
  • Alert B (Critical): Ransomware signature - immediate threat requiring containment
  • Alert C (Medium): Unusual DNS query - potential C2 communication, needs investigation
  • Alert D (Critical): Data exfiltration - active breach, immediate response needed
Correct Priority Order:
  1. Alert B (Ransomware) - Immediate containment prevents spread
  2. Alert D (Data Exfiltration) - Active data loss occurring
  3. Alert C (Suspicious DNS) - Could indicate ongoing compromise
  4. Alert A (Failed SSH) - Lowest risk, single failed attempt
Key Factors: Impact to business operations, active vs. potential threat, speed of escalation needed, data sensitivity involved

Grading Rubric

Correct severity classifications (4 alerts)40 pts
Correct priority ordering30 pts
Clear justification of decisions20 pts
Professional documentation10 pts
2 False Positive Identification Beginner ⏱ 25 min ▼

Learning Objectives

  • Distinguish between true positive and false positive alerts
  • Understand common causes of false positives in SOC environments
  • Learn when to tune detection rules vs. escalate alerts

Screens Used

Screen 1: SIEM Dashboard Screen 4: Threat Intel

Scenario

Your SOC has been receiving numerous alerts that appear to be false positives, creating alert fatigue. Management has asked you to analyze a sample of alerts and recommend which ones might need detection rule tuning.

Tasks

  • Review the following alerts and classify each as likely True Positive (TP) or False Positive (FP):

    Alert 1: Port scan detected from 10.0.5.100 (IT Security Scanner)

    Alert 2: Mimikatz signature detected in memory on DC-01

    Alert 3: Large file download from marketing@company.com (Google Drive sync)

    Alert 4: PowerShell encoded command execution on WKS-089

    Alert 5: Failed authentication from admin service account (nightly backup job)

  • For each False Positive identified, suggest a tuning recommendation
  • For each True Positive, describe the next investigation step
  • Check Screen 4 Threat Intel feed - does any intelligence correlate with the true positives?

💡 Tips for Identifying False Positives

  • Check if the source is a known authorized security tool
  • Look for patterns that match legitimate business processes
  • Verify if activity occurs during expected maintenance windows
  • Cross-reference with asset inventory and user roles
🔒 View Answer Key ▼
Classifications:
  • Alert 1 - FALSE POSITIVE: Known IT security scanner. Tune: Whitelist source IP 10.0.5.100 for port scan rules
  • Alert 2 - TRUE POSITIVE: Mimikatz on Domain Controller is critical. Next step: Immediately isolate DC-01, check for lateral movement
  • Alert 3 - FALSE POSITIVE: Authorized cloud storage sync. Tune: Create exception for approved cloud services
  • Alert 4 - TRUE POSITIVE: Encoded PowerShell is suspicious. Next step: Decode command, check process tree, review user activity
  • Alert 5 - FALSE POSITIVE: Scheduled job authentication. Tune: Add time-based exception for backup window
Key Learning: False positives often come from authorized tools, scheduled jobs, and legitimate business processes. True positives typically involve credential theft tools, encoded commands, and activity on sensitive systems.

Grading Rubric

Correct TP/FP classifications40 pts
Appropriate tuning recommendations25 pts
Correct investigation steps for TPs25 pts
Threat intel correlation attempt10 pts

4 Escalation Decision Making Beginner ⏱ 20 min ▼

Learning Objectives

  • Apply escalation criteria correctly
  • Understand the difference between Tier 1 and Tier 2 responsibilities
  • Document alerts properly for escalation

Screens Used

Screen 1: SIEM Dashboard Screen 3: Incident Response

Scenario

As a Tier 1 analyst, you must decide which alerts to handle yourself and which to escalate to Tier 2. Your SOC's escalation policy states that Tier 1 handles: single failed logins, low-severity malware (PUPs), and informational alerts. Tier 2 handles: confirmed malware, multiple related alerts, and anything affecting critical assets.

Tasks

  • For each scenario below, decide: Handle (Tier 1), Escalate (Tier 2), or Escalate (Tier 3):

    Scenario A: PUP detected on intern's workstation - adware toolbar

    Scenario B: 5 failed logins in 10 minutes from same user account

    Scenario C: Emotet trojan detected on Finance department laptop

    Scenario D: Suspicious process on Domain Controller with unknown binary

    Scenario E: Single informational alert - Windows update check

  • For each escalation, write a brief escalation note (2-3 sentences) that Tier 2 would need
  • Open Screen 3 (Incident Response) and familiarize yourself with the incident queue format
🔒 View Answer Key ▼
Escalation Decisions:
  • Scenario A - HANDLE (Tier 1): PUP/adware is low severity, standard removal
  • Scenario B - ESCALATE (Tier 2): Multiple failed logins suggests brute force attempt
  • Scenario C - ESCALATE (Tier 2): Emotet is confirmed dangerous malware affecting critical department
  • Scenario D - ESCALATE (Tier 3): Unknown binary on DC is potential APT/critical threat
  • Scenario E - HANDLE (Tier 1): Informational only, no action required
Sample Escalation Notes:

Scenario B: "Escalating potential brute force attack. User account [name] has 5 failed login attempts within 10 minutes from IP [x.x.x.x]. Pattern suggests automated attack. Recommend account lockout pending investigation."

Scenario C: "Escalating confirmed Emotet detection on FIN-WKS-012 (Finance Dept). This is a known banking trojan with lateral movement capabilities. Immediate containment recommended due to access to financial systems."

6 Brute Force Attack Investigation Intermediate ⏱ 35 min ▼

Learning Objectives

  • Investigate authentication-based attacks
  • Correlate alerts across multiple data sources
  • Execute containment actions appropriately

Screens Used

Screen 1: SIEM Dashboard Screen 2: Network Monitor Screen 3: Incident Response

Scenario

A Tier 1 analyst has escalated the following alert cluster to you. Multiple failed authentication attempts have been detected targeting the VPN gateway, followed by a successful login.

Alert Time: 08:15:00 - 08:47:00 UTC

Target: VPN Gateway (vpn.company.com)

Source IP: 185.220.101.45 (Tor Exit Node)

Targeted Accounts: admin, administrator, jsmith, mwilliams, svc_backup

Failed Attempts: 347 attempts across 5 accounts

Successful Login: 08:47:23 - Account: svc_backup

Tasks

  • Analyze the attack pattern:
    • What type of brute force attack is this (credential stuffing, password spraying, dictionary)?
    • Why is a Tor exit node significant?
  • Using Screen 2, check if the source IP (185.220.101.45) appears in the Top Talkers or Firewall Logs
  • On Screen 3, create a mental incident workflow:
    • What containment action should happen FIRST?
    • What IOCs should be documented?
    • What systems might the compromised service account access?
  • Write an incident summary documenting: Timeline, Attack Vector, Impact, Containment Actions Taken

MITRE ATT&CK Mapping

T1110.001 - Password Guessing T1078 - Valid Accounts T1133 - External Remote Services
🔒 View Answer Key ▼
Attack Pattern Analysis:
  • Attack Type: Password Spraying - Multiple accounts targeted with likely common passwords (347 attempts across 5 accounts = ~70 attempts each)
  • Tor Significance: Attacker is hiding their true origin; indicates sophistication and likely malicious intent. Legitimate users don't typically VPN through Tor.
Immediate Containment Actions (in order):
  1. Disable/reset svc_backup account immediately
  2. Block source IP 185.220.101.45 at firewall
  3. Terminate any active VPN sessions for svc_backup
  4. Reset passwords for all targeted accounts
  5. Review svc_backup activity since 08:47:23
IOCs to Document:
  • Source IP: 185.220.101.45
  • Compromised Account: svc_backup
  • Attack timeframe: 08:15 - 08:47 UTC
  • Entry point: VPN Gateway

Learning Objectives

Screens Used

Screen 1: SIEM Dashboard Screen 3: Incident Response Screen 4: Threat Intel

Scenario

CRITICAL ALERT: Multiple endpoints are reporting malware detection within minutes of each other. This appears to be a spreading infection.

09:15:33 WKS-042 - TrickBot detected

09:17:45 WKS-089 - TrickBot detected

09:18:02 WKS-042 - Lateral movement attempt to FILE-SVR-01

09:19:11 WKS-103 - TrickBot detected

09:20:30 FILE-SVR-01 - Suspicious SMB activity

09:21:45 DC-01 - Authentication anomaly detected

Hash (SHA256): e99a18c428cb38d5f260853678922e03abd833

Tasks

  • On Screen 4, search the IOC Feed for TrickBot intelligence. What additional capabilities should you expect?
  • Create an isolation priority list - which systems should be isolated first and why?
  • Using Screen 3 as reference, determine which response actions are appropriate:
    • Should you isolate individual hosts or segment the network?
    • Should DC-01 be isolated? What are the risks either way?
  • Document affected assets and their roles
  • Write an executive summary (suitable for CISO briefing) explaining the situation

MITRE ATT&CK Mapping

T1566 - Phishing T1055 - Process Injection T1021.002 - SMB/Windows Admin Shares T1003 - Credential Dumping
🔒 View Answer Key ▼
TrickBot Capabilities:
  • Banking credential theft
  • Lateral movement via SMB
  • Credential harvesting (Mimikatz-like)
  • Often precursor to Ryuk/Conti ransomware
  • Persistence mechanisms
Isolation Priority (in order):
  1. FILE-SVR-01 - Prevent further lateral movement and data access
  2. WKS-042 - Patient zero, most likely to have full infection
  3. WKS-089, WKS-103 - Confirmed infected workstations
  4. DC-01 - Monitor closely but DON'T isolate yet (would disrupt all authentication)
DC-01 Decision:

Do NOT immediately isolate DC-01 - Isolating a Domain Controller stops all domain authentication, effectively halting business operations. Instead: Increase monitoring, disable the anomalous account, prepare offline backup of AD, have isolation ready as last resort if active compromise confirmed.

Executive Summary Example:

"At 09:15 today, we detected a TrickBot malware outbreak affecting at least 3 workstations with attempted lateral movement to our file server. TrickBot is known to precede ransomware deployment. We have isolated affected workstations and the file server. Our Domain Controller shows suspicious activity and is under enhanced monitoring. Recommended actions: Authorize network-wide scan, prepare communications for potential broader containment, and brief legal/PR for possible disclosure requirements."

MITRE ATT&CK Mapping

T1048 - Exfiltration Over Alternative Protocol T1567 - Exfiltration Over Web Service T1041 - Exfiltration Over C2 Channel
🔒 View Answer Key ▼
Why This is Suspicious (vs. Legitimate):
  • Timing: 02:30-03:15 is outside business hours with no user sessions
  • Destination: Unknown VPS, not a corporate backup destination
  • Source: Database server shouldn't initiate outbound transfers
  • Volume: 2.3GB from DB server = potentially millions of records
  • Protocol: HTTPS to cloud VPS suggests staging for extraction
Potential Data Exposed:
  • Customer PII (names, addresses, emails)
  • Payment information (if stored)
  • Account credentials
  • Transaction history
Compliance Implications:
  • GDPR: 72-hour notification requirement if EU customers affected
  • CCPA: Notification to California residents
  • PCI-DSS: If payment data involved, immediate card brand notification
  • Potential regulatory fines and mandatory breach disclosure
Immediate Actions:
  1. Block destination IP at firewall
  2. Isolate DB-SVR-02 from network
  3. Preserve all logs and memory forensics
  4. Notify Legal and Privacy Officer
  5. Begin forensic investigation of DB server

Your organization has been mentioned in a threat intelligence report as a potential target of APT41 (Double Dragon). You need to proactively hunt for indicators of compromise.

Target Industries: Healthcare, Telecom, Technology, Gaming

C2 Domains: update-service[.]net, cdn-microsoft[.]com

Tasks

    • Search for web shell indicators (aspx/php in unusual locations)
  • Using Screen 2 (Network), look for:
    • Beaconing patterns (regular interval callbacks)
    • Unusual SSL certificates on internal hosts
    • Traffic to the specified geographic regions
  • Using Screen 4 (Threat Intel), cross-reference any findings with known APT41 campaigns
  • Document your hunt: hypothesis, data sources queried, findings, and conclusions

MITRE ATT&CK - APT41 Common Techniques

T1195 - Supply Chain Compromise T1505.003 - Web Shell T1071 - Application Layer Protocol T1053 - Scheduled Task
🔒 View Answer Key ▼
Sample Threat Hunting Hypothesis:

"Given that APT41 targets organizations in our industry using spearphishing and web shells, we hypothesize that if we are compromised, we would see: (1) DNS queries to typosquat Microsoft/update domains, (2) web shell files in public-facing web directories, (3) scheduled tasks executing suspicious PowerShell, and (4) beaconing traffic to Asian IP ranges during off-hours."

Hunt Data Sources:
  • DNS logs - Query for *update-service* and *cdn-microsoft*
  • Firewall logs - Connections to 103.224.182.0/24
  • Web server logs - New .aspx/.php files in web roots

  • NetFlow - Beaconing pattern detection
Beaconing Indicators:
  • Regular interval connections (e.g., every 60 seconds)
  • Small data transfers (check-in packets)
  • HTTPS to non-standard ports
  • Connections during non-business hours
Hunt Documentation Format:
  1. Hypothesis Statement
  2. Data Sources Queried
  3. Queries/Searches Performed
  4. Findings (with evidence)
  5. Conclusion (confirmed/not confirmed)

Scenario

A critical zero-day vulnerability has been publicly disclosed and is being actively exploited in the wild. No patch is currently available.

CVE: CVE-2024-XXXX

Affected: Apache Web Server 2.4.x (all versions)

CVSS: 9.8 (Critical)

Type: Remote Code Execution

Exploitation: Active exploitation confirmed

Attack Vector: Malformed HTTP headers

Patch Status: No patch available - ETA 48-72 hours

Tasks

  • Using Screen 4 Endpoint Status, identify how many Apache servers exist in your environment
  • Check Screen 4 Vulnerability Management panel - would this CVE appear there?
  • Develop a risk assessment:
    • Which systems are internet-facing vs. internal only?
    • What is the business impact if these servers are compromised?

    • Network-level mitigations
    • WAF rules that could help
    • Monitoring/detection signatures
  • Write a CISO briefing on exposure and mitigation status
🔒 View Answer Key ▼
Inventory Assessment Approach:
  • Search endpoint inventory for "Apache" or "httpd" processes
  • Check asset management for servers with web server role
  • Query network scans for port 80/443 services
  • Review deployment documentation for web infrastructure
Risk Assessment Framework:
  • Critical: Internet-facing Apache serving customer portal
  • High: Internet-facing Apache serving marketing site
  • Medium: Internal Apache for development tools
  • Low: Internal Apache on isolated network segments
Compensating Controls:

  • Detection: Alert on Apache crash/restart, monitor for webshell indicators, watch for unusual child processes
